# Участие в разработке

Спасибо за интерес к LiteEditorAI! Проект на стадии **alpha** — баги, идеи и pull request'ы приветствуются.

## Баг-репорты и идеи

Откройте [Issue](https://github.com/DanielLetto2020/LiteEditorAI/issues): что ожидали, что произошло,
ОС и версия (видна в меню «О программе»), по возможности — скриншот или вывод DevTools (`F12`).

## Сборка и запуск

```bash
npm install        # зависимости + сборка node-pty под Electron (postinstall)
npm start          # собрать фронт и запустить
npm run watch      # esbuild в watch-режиме для фронта
```

Требуется Node.js 22+ (x64, Linux или Windows).

## Архитектура (кратко)

- **`main.js`** — тонкий Electron-бэкенд: PTY, файловые операции, git, окно. НЕ бандлится.
- **`preload.js`** — мост `window.lite` через `contextBridge` (contextIsolation вкл., nodeIntegration выкл.).
- **`renderer/renderer.js`** — вся логика UI, бандлится esbuild (`node build.js`).
- Состояние — в `~/.LiteEditorAI/` (проекты, настройки, заметки).

Правки `main.js`/`preload.js` применяются на следующем запуске; правки `renderer.js` требуют пересборки
(`npm start` пересобирает сам).

## Pull requests

Прямой доступ к репозиторию (write/collaborator) не выдаётся и **не требуется** — участие идёт через форк:

1. **Сделайте форк** репозитория и создайте ветку от ветки **`contrib`**.
2. Внесите правки. Небольшие сфокусированные PR проще ревьюить.
3. Откройте **pull request с базовой веткой `contrib`** — **не** в `main` и **не** в `dev`.

Почему именно `contrib`: `main` — это релизное «лицо» (формируется снапшотом при каждом релизе, прямые
коммиты в неё затираются), а ветка разработки приватная. **`contrib`** — публичная ветка, специально
предназначенная для приёма pull request'ов; принятые правки переносятся во внутреннюю ветку разработки и
выходят в одном из ближайших релизов.

При оформлении PR:

- **Стиль** — как в окружающем коде (линтера в проекте нет). Язык интерфейса и комментариев — как в проекте.
- **Не меняйте номер версии** (`package.json` / `APP_VERSION`) и релизные файлы (`CHANGELOG.md`,
  `RELEASE_NOTES.md`) — версионированием и релизами занимается мейнтейнер.
- **Никакого хардкода** цветов/теней/радиусов — только CSS-токены (`var(--…)`).
- Тестов в проекте нет: перед PR прогоните сборку (`node build.js`) и проверьте изменения вручную.

**Как проходит ревью.** Мейнтейнер — единственный, кто вливает изменения. Pull request'ы рассматриваются и
переносятся в разработку **вручную, по усмотрению мейнтейнера и без фиксированных сроков**; решение о том,
нужен ли тот или иной PR, остаётся за мейнтейнером. Это нормальная практика — не расстраивайтесь, если PR
обсуждается, дорабатывается или отклоняется.
